# 手机号加密方案

## 一、明文存储存在以下问题

用户的手机号存储采用的是最原始的明文存储，这会有存在极大的风险性问题。假如某天有人故意将这份数据导出到外边拿去贩卖，那就是个很大的问题。

## 二、业内常用的加密处理策略

### 对称加密算法（加密解密用同一密钥）

对称加密的核心特点是效率极高（计算量小），适合短数据、高频次加解密场景，是手机号 / 邮箱加密的主流选择。

| 算法        | 密钥长度                 | 安全性                     | 效率               | 可逆性 | 适用场景 |
| ----------- | ------------------------ | -------------------------- | ------------------ | ------ | -------- |
| AES（推荐） | 128/192/256 位           | 极高（256 位可抗量子攻击） | 极快               | 是     | 短数据   |
| DES         | 56 位                    | 极低（已被暴力破解）       | 较快               | 是     | 历史遗留 |
| 3DES        | 168 位（3 个 56 位密钥） | 中（安全性低于 AES）       | 较慢（AES 的 1/3） | 是     | 需兼容   |

### 非对称加密算法（公钥加密、私钥解密）

非对称加密的核心特点是安全性高（密钥分离，公钥可公开），但效率极低（计算量是对称加密的 100-1000 倍），适合小数据量、低频次场景。

| 算法 | 密钥长度      | 安全性 | 效率                   | 可逆性 |
| ---- | ------------- | ------ | ---------------------- | ------ |
| RSA  | 2048 位及以上 | 高     | 慢                     | 是     |
| ECC  | 256 位        | 极高   | 中（比 RSA 快 3-5 倍） | 是     |

结合手机号 / 邮箱的短数据、高频次、需还原的特点，我们可以尝试使用 AES 算法来实现。

## 三、结论

**AES 算法（推荐 AES-128-GCM 模式）**

理由：效率最高（微秒级处理 11 位手机号），安全性符合国际标准（NIST 认证），GCM 模式自带"加密 + 完整性校验"（防止数据被篡改），且支持轻量化实现（适合服务器 / 移动端）。每次生成的结果都会随机变化，难以破解。

注意：需严格管理对称密钥（如用密钥管理系统 KMS 存储，定期轮换）。

## 四、加密数据在数据库层如何做检索

### 利用数据库内置加解密函数

在 MySQL 存储层其实也有内置支持一些专门用于加解密使用的特殊函数，例如 MD5，DES，AES 等。我们可以对 SQL 进行修改成如下案例：

```sql
select id, AES_decrypt(phone,'密钥') from t_user_phone;
```

ps：MySQL 内部的 AES_decrypt 函数默认返回的是二进制格式的数据，因此底层存储需要用 blob 或者 VARBINARY 格式来存储。

**==性能极差，不推荐。会导致全表扫描，对于大数据量存储的场景不适合==。**

如果足够厉害，可以对 MySQL 的底层源代码直接做修改，自定义一套加解密数据的检索能力内置进去，不过这样做的开发难度极高。而且聘请一位懂这方面技术的专家的成本也是极高的。

### 在应用层做加解密处理

可以通过 MyBatis 框架，给我们的手机号，邮件这样的字段增加一个注解，然后在 ORM 框架层做注解判断的拦截处理。如果命中，则利用自定义的算法工具做加解密。DB 层依然存储的是密文本身，然后解密的密钥通过应用层配置来控制。这样即使说数据脱库了，但是操作者不知道密钥，依然无法解密。

> **对于精确查询可以支持，但是模糊查询类型不好支持。**

### 分段式存储

还是继续基于应用层去做加解密式的处理，但是我们对于底层 DB 的存储做一些扩展。例如手机号为：`099 784 927`，我们将其拆成：099，784，927，然后利用工具加密后的密文假设为：

| 号段内容 | 加密后         |
| -------- | -------------- |
| 099      | H923hbyubqwfty |
| 784      | 89h3iuFZsHwe3a |
| 927      | QwGkagj8h7834G |

然后 DB 层将这些分段后的加密内容一并保存起来。

```sql
insert into t_user_phone (encrypt_phone_str,encrypt_block) values 
('z6EDHN5400Ht+TbIJzC6GlnlYVWKx4coQngcynBPWhXU6Ev0rYSM','H923hbyubqwfty,89h3iuFZsHwe3a,QwGkagj8h7834G');
```

这样后续做模糊搜索能力的时候，我们可以提供出基于加密后数据段的简单检索能力。

例如：按照手机的前三位或者手机号的末尾三位号码做检索。

```sql
# 前三位手机号加密后的模糊检索
select * from t_user_phone where encrypt_block like 'H923hbyubqwfty,%';

# 后三位手机号加密后的模糊检索
select * from t_user_phone where encrypt_block like '%,QwGkagj8h7834G';
```

==**可以应付模糊检索的能力，但是只能解决部分场景下的模糊检索功能，只适合用于简单加密逻辑场景中，如果底层是类似于 AES 这种高强度的加密算法的话，分段存储思路不适合。**==

## 五、小结

目前小绿书阶段还没有需要做到手机号这些数据做模糊查询的场景，分段式存储的方案可以保留，暂时只需要做到应用层基于 ORM 框架实现加解密逻辑处理即可。
