# 文章创作流程分析

## 🚀 AI 文章生成器 — 完整代码执行流程详解

整个应用的核心目标是：**用户输入一个主题，AI 分阶段生成一篇图文并茂的文章，用户在每个阶段可以参与决策。**

## 📋 整体架构概览

```
用户请求 → Controller → Service(业务逻辑) → AsyncService(异步编排) → AgentService(智能体调用LLM) → SSE推送结果
```

应用采用 **三阶段、六智能体** 的流水线式架构：

| 阶段  | 智能体    | 功能                                | 用户参与       |
| ----- | --------- | ----------------------------------- | -------------- |
| 阶段1 | 智能体1   | 生成3-5个标题方案                   | ✅ 用户选择标题 |
| 阶段2 | 智能体2   | 根据选定标题生成大纲                | ✅ 用户编辑大纲 |
| 阶段3 | 智能体3-6 | 生成正文→分析配图→生成配图→图文合成 | ❌ 自动完成     |



## 🔍 第一步：用户发起创建文章请求

**入口**：ArticleController.java 的 `POST /article/create`

```java
@PostMapping("/create")
public BaseResponse<String> createArticle(@RequestBody ArticleCreateRequest request, ...) 
```

**请求体** [ArticleCreateRequest](file:///d:/TPROJECT/code/ai-passage-creator-main/passage-backend/src/main/java/com/yupi/template/model/dto/article/ArticleCreateRequest.java) 包含三个字段：
- `topic`：选题（必填）
- `style`：文章风格（可选，如 tech/emotional/educational/humorous）
- `enabledImageMethods`：允许的配图方式列表（可选，如 PEXELS、NANO_BANANA 等）

**为什么这么设计？** 让用户能控制文章的调性和配图方式，而不是完全由 AI 决定，提高生成结果的可用性。



## 🔍 第二步：参数校验 + 配额检查 + 创建任务

Controller 中做了几件关键的事：

**1. 参数校验**
```java
ThrowUtils.throwIf(request.getTopic() == null || request.getTopic().trim().isEmpty(), 
        ErrorCode.PARAMS_ERROR, "选题不能为空");
ThrowUtils.throwIf(!ArticleStyleEnum.isValid(request.getStyle()),
        ErrorCode.PARAMS_ERROR, "无效的文章风格");
```
确保选题不为空、风格值合法，防止无效请求进入系统。

**2. 获取登录用户**
```java
User loginUser = userService.getLoginUser(httpServletRequest);
```
从 Session/Token 中获取当前用户，用于后续的权限校验和配额扣减。

**3. 配额检查 + 创建任务（事务内）**
```java
String taskId = articleService.createArticleTaskWithQuotaCheck(
        request.getTopic(), request.getStyle(), request.getEnabledImageMethods(), loginUser);
```

进入 [ArticleServiceImpl](file:///d:/TPROJECT/code/ai-passage-creator-main/passage-backend/src/main/java/com/yupi/template/service/impl/ArticleServiceImpl.java)，这个方法用 `@Transactional` 保证**原子性**：

```java
@Transactional(rollbackFor = Exception.class)
public String createArticleTaskWithQuotaCheck(...) {
    quotaService.checkAndConsumeQuota(loginUser);  // 先扣配额
    return createArticleTask(topic, style, enabledImageMethods, loginUser);  // 再创建任务
}
```

**为什么要在同一事务中？** 如果先扣配额再创建任务，但任务创建失败，配额就白扣了。事务保证要么都成功，要么都回滚。

`createArticleTask` 内部做了：
- 处理配图方式权限（普通用户不能用 VIP 专属的 NANO_BANANA 和 SVG_DIAGRAM）
- 生成 UUID 作为 taskId
- 创建 Article 实体，初始状态为 `PENDING`，初始阶段为 `PENDING`
- 保存到数据库

**4. 异步启动阶段1**
```java
articleAsyncService.executePhase1(taskId, request.getTopic(), request.getStyle());
```
立即返回 taskId 给前端，不等待 AI 生成完成。

**为什么异步？** AI 生成标题需要几秒到十几秒，如果同步等待，HTTP 请求会超时，用户体验极差。异步让接口秒级返回。



## 🔍 第三步：SSE 连接建立（前端订阅进度）

**入口**：`GET /article/progress/{taskId}`

前端拿到 taskId 后，立即建立 SSE（Server-Sent Events）长连接：

```java
@GetMapping("/progress/{taskId}")
public SseEmitter getProgress(@PathVariable String taskId, ...) {
    // 权限校验
    articleService.getArticleDetail(taskId, loginUser);
    // 创建 SSE Emitter
    SseEmitter emitter = sseEmitterManager.createEmitter(taskId);
    return emitter;
}
```

[SseEmitterManager](file:///d:/TPROJECT/code/ai-passage-creator-main/passage-backend/src/main/java/com/yupi/template/manager/SseEmitterManager.java) 用 `ConcurrentHashMap` 管理所有活跃的 SSE 连接：

```java
private final Map<String, SseEmitter> emitterMap = new ConcurrentHashMap<>();
```

**为什么用 SSE 而不是 WebSocket？** SSE 是单向推送（服务器→客户端），更适合"进度通知"场景，实现更简单，且天然支持 HTTP 协议。WebSocket 适合双向通信，但这里不需要客户端主动发消息。



## 🔍 第四步：阶段1 — 生成标题方案

**入口**：[ArticleAsyncService.executePhase1](file:///d:/TPROJECT/code/ai-passage-creator-main/passage-backend/src/main/java/com/yupi/template/service/ArticleAsyncService.java)

```java
@Async("articleExecutor")  // 在专用线程池中执行
public void executePhase1(String taskId, String topic, String style) {
```

**为什么用 `@Async("articleExecutor")`？** 使用自定义线程池（核心5线程，最大10线程），避免占用 Tomcat 的 HTTP 线程，同时控制并发数防止 AI API 被打爆。

执行流程：

1. **更新状态**：`PENDING → PROCESSING`，阶段更新为 `TITLE_GENERATING`
2. **创建状态对象** `ArticleState`：这是贯穿整个流程的**共享状态对象**，所有智能体的输入输出都通过它传递
3. **选择执行模式**：

```java
if (useOrchestrator) {
    articleAgentOrchestrator.executePhase1_GenerateTitles(state, ...);
} else {
    articleAgentService.executePhase1_GenerateTitles(state, ...);
}
```

系统支持两种模式：
- **原有模式**（`ArticleAgentService`）：直接调用各智能体方法
- **多智能体编排模式**（`ArticleAgentOrchestrator`）：使用 Spring AI Alibaba 的 StateGraph 编排，支持更复杂的流程控制

**智能体1 做了什么？** 进入 `agent1GenerateTitleOptions`：

```java
String prompt = PromptConstant.AGENT1_TITLE_PROMPT
        .replace("{topic}", state.getTopic())    // 替换选题
        + getStylePrompt(state.getStyle());        // 追加风格指令

String content = callLlm(prompt);  // 调用大模型（通义千问/DashScope）
```

**Prompt 的设计**（[PromptConstant](file:///d:/TPROJECT/code/ai-passage-creator-main/passage-backend/src/main/java/com/yupi/template/constant/PromptConstant.java)）：

```
你是一位爆款文章标题专家...
根据以下选题,生成 3-5 个爆款文章标题方案:
选题：{topic}
要求: 主标题包含数字、情绪化词汇...
请直接返回 JSON 格式...
```

**为什么这么设计 Prompt？**
- **角色设定**（"爆款文章标题专家"）：让 LLM 进入特定角色，输出更专业
- **明确数量**（3-5个）：给 AI 约束，避免输出过多或过少
- **要求 JSON 格式**：方便程序解析，避免自由文本难以结构化
- **风格追加**：如果用户选了"科技风格"，会追加"语言专业、严谨，多使用专业术语"等指令

**调用 LLM 的方式**：

```java
private String callLlm(String prompt) {
    ChatResponse response = chatModel.call(new Prompt(new UserMessage(prompt)));
    return response.getResult().getOutput().getText();
}
```

使用 Spring AI 的 `DashScopeChatModel`（阿里通义千问），同步调用，等待完整响应。

**解析结果**：

```java
List<ArticleState.TitleOption> titleOptions = parseJsonListResponse(
        content, new TypeToken<>(){}, "标题方案");
state.setTitleOptions(titleOptions);
```

将 LLM 返回的 JSON 解析为 `List<TitleOption>`，每个包含 `mainTitle` 和 `subTitle`。

4. **保存到数据库** + **更新阶段**为 `TITLE_SELECTING`
5. **通过 SSE 推送**标题方案给前端：

```java
Map<String, Object> data = new HashMap<>();
data.put("titleOptions", state.getTitleOptions());
sendSseMessage(taskId, SseMessageTypeEnum.TITLES_GENERATED, data);
```

前端收到 `TITLES_GENERATED` 消息后，展示标题选项让用户选择。



## 🔍 第五步：用户确认标题 → 阶段2 生成大纲

**入口**：`POST /article/confirm-title`

```java
@PostMapping("/confirm-title")
public BaseResponse<Void> confirmTitle(@RequestBody ArticleConfirmTitleRequest request, ...) {
```

请求包含：`taskId`、`selectedMainTitle`、`selectedSubTitle`、`userDescription`（用户补充描述，可选）

**为什么允许用户补充描述？** 用户可能想对文章方向做额外指引，比如"重点讲实战案例"、"面向初学者"等，这些信息会影响大纲生成。

`confirmTitle` 方法做了：
- 权限校验
- **阶段校验**：必须是 `TITLE_SELECTING` 才能操作，防止乱序调用
- 保存用户选择的标题和补充描述
- 更新阶段为 `OUTLINE_GENERATING`

然后异步启动阶段2：
```java
articleAsyncService.executePhase2(request.getTaskId());
```

**智能体2 做了什么？** 进入 `agent2GenerateOutline`：

```java
String descriptionSection = "";
if (state.getUserDescription() != null && !state.getUserDescription().trim().isEmpty()) {
    descriptionSection = PromptConstant.AGENT2_DESCRIPTION_SECTION
            .replace("{userDescription}", state.getUserDescription());
}

String prompt = PromptConstant.AGENT2_OUTLINE_PROMPT
        .replace("{mainTitle}", ...)
        .replace("{subTitle}", ...)
        .replace("{descriptionSection}", descriptionSection)  // 动态插入用户描述
        + getStylePrompt(state.getStyle());
```

**注意这里用了流式输出**：

```java
String content = callLlmWithStreaming(prompt, streamHandler, SseMessageTypeEnum.AGENT2_STREAMING);
```

```java
private String callLlmWithStreaming(String prompt, Consumer<String> streamHandler, SseMessageTypeEnum messageType) {
    StringBuilder contentBuilder = new StringBuilder();
    Flux<ChatResponse> streamResponse = chatModel.stream(new Prompt(new UserMessage(prompt)));
    streamResponse
            .doOnNext(response -> {
                String chunk = response.getResult().getOutput().getText();
                if (chunk != null && !chunk.isEmpty()) {
                    contentBuilder.append(chunk);
                    streamHandler.accept(messageType.getStreamingPrefix() + chunk);
                }
            })
            .blockLast();
    return contentBuilder.toString();
}
```

**为什么大纲和正文用流式输出？** 
- 大纲和正文的生成时间较长（10-30秒），流式输出让前端可以实时显示生成过程
- 用户体验更好，不会感觉"卡住了"
- 流式消息格式为 `AGENT2_STREAMING:内容片段`，前端按前缀识别类型

生成完成后，解析 JSON 为 `OutlineResult`（包含 `sections` 列表，每个 section 有序号、标题、要点），保存到数据库，阶段更新为 `OUTLINE_EDITING`，SSE 推送 `OUTLINE_GENERATED`。



## 🔍 第六步：用户确认大纲 → 阶段3 生成正文+配图

**入口**：`POST /article/confirm-outline`

用户可以编辑大纲（增删改章节），确认后触发阶段3。

**阶段3 是最复杂的，包含4个智能体顺序执行：**

#### 智能体3：生成正文

```java
String outlineText = GsonUtils.toJson(state.getOutline().getSections());
String prompt = PromptConstant.AGENT3_CONTENT_PROMPT
        .replace("{mainTitle}", ...)
        .replace("{subTitle}", ...)
        .replace("{outline}", outlineText)    // 把大纲 JSON 喂给 AI
        + getStylePrompt(state.getStyle());
```

Prompt 要求 AI 根据大纲创作正文，每章节300-400字，使用 Markdown 格式。同样使用流式输出。

**为什么要把大纲 JSON 传给 AI？** 让 AI 严格按照大纲结构写正文，而不是自由发挥，保证文章结构和用户确认的一致。

#### 智能体4：分析配图需求

这是最精巧的智能体，它做两件事：

1. **在正文中插入图片占位符**：如 `{{IMAGE_PLACEHOLDER_1}}`、`{{ICON_PLACEHOLDER_1}}`
2. **生成配图需求列表**：每个需求包含位置、类型、图片来源、关键词/提示词

```java
String prompt = PromptConstant.AGENT4_IMAGE_REQUIREMENTS_PROMPT
        .replace("{mainTitle}", ...)
        .replace("{content}", state.getContent())           // 正文
        .replace("{availableMethods}", availableMethods)     // 可用配图方式
        .replace("{methodUsageGuide}", methodUsageGuide);    // 各方式使用指南
```

**为什么用占位符方案？** 
- 先生成正文，再分析哪里需要配图，比一开始就混写更灵活
- 占位符 `{{IMAGE_PLACEHOLDER_N}}` 可以精确定位图片插入位置
- 不同类型的占位符（IMAGE vs ICON）支持不同的排版方式

**配图方式有6种**（策略模式）：

| 方式        | 说明         | 适用场景      |
| ----------- | ------------ | ------------- |
| PEXELS      | 图库搜索     | 真实照片      |
| NANO_BANANA | AI 生图      | 创意插画      |
| MERMAID     | 代码生成图表 | 流程图/架构图 |
| ICONIFY     | 图标库       | 小图标        |
| EMOJI_PACK  | 表情包       | 轻松幽默      |
| SVG_DIAGRAM | AI 生成 SVG  | 概念示意图    |

**为什么用策略模式？** 不同配图方式的获取逻辑完全不同（有的搜图库、有的调 AI、有的生成代码），策略模式让每种方式独立实现，互不干扰，且方便扩展新的配图方式。

智能体4 还会**验证配图需求**：

```java
List<ArticleState.ImageRequirement> validatedRequirements = validateAndFilterImageRequirements(
        agent4Result.getImageRequirements(), state.getEnabledImageMethods());
```

确保 AI 不会选择用户未授权的配图方式（比如普通用户不能用 VIP 的 NANO_BANANA），如果选了就降级替换。

#### 智能体5：生成配图

遍历每个配图需求，使用策略模式获取图片：

```java
for (ArticleState.ImageRequirement requirement : state.getImageRequirements()) {
    ImageServiceStrategy.ImageResult result = imageServiceStrategy.getImageAndUpload(imageSource, imageRequest);
    // ...
}
```

[ImageServiceStrategy](file:///d:/TPROJECT/code/ai-passage-creator-main/passage-backend/src/main/java/com/yupi/template/service/ImageServiceStrategy.java) 的 `getImageAndUpload` 方法：

1. 根据 `imageSource` 找到对应的 `ImageSearchService` 实现
2. 调用 `getImageData()` 获取图片数据
3. 上传到 COS（腾讯云对象存储）
4. 返回 COS URL

**为什么统一上传到 COS？**
- 外部图片 URL 可能失效（如 Pexels 链接可能过期）
- COS URL 稳定可靠，加载速度快
- 统一管理所有图片资源

**降级机制**：如果任何配图方式失败，自动使用 Picsum 随机图片作为兜底：

```java
if (service == null || !service.isAvailable()) {
    return handleFallbackWithUpload(request.getPosition());
}
```

**为什么需要降级？** 外部服务（Pexels API、AI 生图 API）可能不可用，降级保证文章总能生成，不会因配图失败导致整个任务失败。

每生成一张配图，通过 SSE 推送 `IMAGE_COMPLETE` 消息，前端可以实时显示配图进度。

#### 智能体6：图文合成

```java
public void mergeImagesIntoContent(ArticleState state) {
    String fullContent = content;
    for (ArticleState.ImageResult image : images) {
        String placeholder = image.getPlaceholderId();
        String imageMarkdown = "![" + image.getDescription() + "](" + image.getUrl() + ")";
        fullContent = fullContent.replace(placeholder, imageMarkdown);
    }
    state.setFullContent(fullContent);
}
```

**为什么最后才合成？** 
- 正文和配图是独立生成的，合成前两者都可以独立修改
- 占位符方案让合成过程非常简单——就是字符串替换
- 如果某张配图失败，只需跳过该占位符，不影响其他内容

articleService.saveArticleContent(taskId, state);

## 🔍 SSE 消息流转全图

整个过程中，SSE 消息按顺序推送：

```
AGENT1_COMPLETE     → 智能体1完成（标题方案生成）
TITLES_GENERATED    → 标题方案已生成，等待用户选择
                      ↑ 用户选择标题 ↑
AGENT2_STREAMING:xx → 大纲流式输出（多条）
AGENT2_COMPLETE     → 智能体2完成
OUTLINE_GENERATED   → 大纲已生成，等待用户编辑
                      ↑ 用户确认大纲 ↑
AGENT3_STREAMING:xx → 正文流式输出（多条）
AGENT3_COMPLETE     → 智能体3完成
AGENT4_COMPLETE     → 智能体4完成（配图需求分析）
IMAGE_COMPLETE      → 单张配图完成（多条）
AGENT5_COMPLETE     → 智能体5完成（配图生成）
MERGE_COMPLETE      → 图文合成完成
ALL_COMPLETE        → 全部完成
```



## 🔍 多智能体编排模式（可选）

[ArticleAgentOrchestrator](file:///d:/TPROJECT/code/ai-passage-creator-main/passage-backend/src/main/java/com/yupi/template/agent/ArticleAgentOrchestrator.java) 使用 Spring AI Alibaba 的 `StateGraph` 编排智能体：

```java
// 阶段3的图：4个节点顺序执行
new StateGraph(keyStrategyFactory)
    .addNode("content_generator", node_async(contentGeneratorAgent))
    .addNode("image_analyzer", node_async(imageAnalyzerAgent))
    .addNode("parallel_image_generator", node_async(parallelImageGenerator))
    .addNode("content_merger", node_async(contentMergerAgent))
    .addEdge(START, "content_generator")
    .addEdge("content_generator", "image_analyzer")
    .addEdge("image_analyzer", "parallel_image_generator")
    .addEdge("parallel_image_generator", "content_merger")
    .addEdge("content_merger", END);
```

**为什么要有两种模式？**
- 原有模式简单直接，容易调试
- 编排模式更灵活，支持并行执行、条件分支、状态检查点等高级特性
- 通过配置 `article.agent.orchestrator.enabled` 切换，方便渐进式迁移

**注意**：同类方法调用时 Spring AOP 代理会失效，所以代码中用了 `AopContext.currentProxy()` 获取代理对象：

```java
private ArticleAgentService getProxy() {

}
```



## 🔍 阶段状态机

[ArticlePhaseEnum](file:///d:/TPROJECT/code/ai-passage-creator-main/passage-backend/src/main/java/com/yupi/template/model/enums/ArticlePhaseEnum.java) 定义了严格的状态转换规则：

```
PENDING → TITLE_GENERATING → TITLE_SELECTING → OUTLINE_GENERATING → OUTLINE_EDITING → CONTENT_GENERATING
```

每个阶段只能转换到下一个阶段，`confirmTitle` 和 `confirmOutline` 方法都会校验当前阶段是否合法。

**为什么需要状态机？** 防止用户乱序操作（比如还没生成标题就去确认大纲），保证流程的确定性。



## 📊 总结：完整数据流

```
用户输入主题
    ↓
Controller: 校验参数 → 检查配额 → 创建Article记录(PENDING) → 返回taskId
    ↓ (异步线程)
AsyncService.executePhase1: 更新状态(PROCESSING) → 调用智能体1 → 保存标题方案 → SSE推送
    ↓ (用户选择标题)
Controller.confirmTitle: 校验阶段 → 保存标题 → 更新阶段
    ↓ (异步线程)
AsyncService.executePhase2: 调用智能体2(流式) → 保存大纲 → SSE推送
    ↓ (用户编辑大纲)
Controller.confirmOutline: 校验阶段 → 保存大纲 → 更新阶段
    ↓ (异步线程)
AsyncService.executePhase3:
    智能体3: 生成正文(流式) → SSE推送
    智能体4: 分析配图需求 → 插入占位符 → SSE推送
    智能体5: 逐张生成配图 → 上传COS → SSE推送
    智能体6: 占位符替换为图片Markdown → SSE推送
    ↓
保存完整文章 → 状态(COMPLETED) → SSE推送ALL_COMPLETE → 关闭连接
```

这就是整个应用从"用户输入主题"到"生成图文并茂的文章"的完整执行过程！核心设计思想是：**人机协作（用户在每个关键节点参与决策）+ 异步流式（SSE 实时反馈）+ 多智能体分工（每个 AI 智能体专注一件事）+ 策略模式（灵活支持多种配图方式）**。
