# Stripe 支付流程 FAQ

## 一、Stripe 怎么知道回调哪个接口？

Stripe 的 Webhook URL 不是在代码里配置的，而是在 **Stripe 后台管理平台** 设置的。

开发者在 Stripe 官网的 Dashboard 中配置：

```
Stripe Dashboard → Developers → Webhooks → Add endpoint
```

```
┌─────────────────────────────────────────────────────────────┐
│  Add a webhook endpoint                                     │
│                                                             │
│  Endpoint URL:                                              │
│  ┌───────────────────────────────────────────────────────┐  │
│  │ https://your-domain.com/webhook/stripe                │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │
│  Select events to listen for:                               │
│  ☑ checkout.session.completed                              │
│  ☑ checkout.session.async_payment_succeeded                 │
│  ☑ checkout.session.expired                                │
│  ...                                                        │
│                                                             │
│                              [Add endpoint]                 │
└─────────────────────────────────────────────────────────────┘
```

**Stripe 会向这个 URL 发送 POST 请求**，告诉你的服务器各种支付事件。

### 📜 代码中给 Stripe 传递的信息

Stripe 需要知道两件事：

**1. 成功/失败后跳转到哪里（前端页面）** —— 这在代码里配置

```java
// PaymentServiceImpl.createStripeSession()
SessionCreateParams params = SessionCreateParams.builder()
    .setSuccessUrl(stripeConfig.getSuccessUrl())    // ← 支付成功后跳转的前端页面
    .setCancelUrl(stripeConfig.getCancelUrl())      // ← 支付取消后跳转的前端页面
    // ...
```

**2. 回调通知发到哪里（Webhook）** —— 这在 Stripe Dashboard 配置

```
Stripe Dashboard 配置:  https://your-domain.com/webhook/stripe
                              ↑
                         开发者手动配置的
```

**Session 和 Webhook 是两个独立的机制**：

| 机制                       | 作用                                            | 配置位置              |
| -------------------------- | ----------------------------------------------- | --------------------- |
| **successUrl / cancelUrl** | 支付完成后，**用户浏览器**跳转到指定页面        | 代码配置              |
| **Webhook**                | 支付完成后，**Stripe 服务器** POST 到你的服务器 | Stripe Dashboard 配置 |

---

### 🔄 两者的工作时机

```
用户支付
    │
    ├── Stripe 服务器处理支付...
    │
    ├── 1. Stripe POST 到 /webhook/stripe  ──► 后端更新用户为 VIP
    │         (Stripe → 你的服务器)
    │
    └── 2. 浏览器跳转到 successUrl  ──► 前端显示"支付成功"
              (Stripe → 用户浏览器 → 你的前端页面)
```

---

### 💡 为什么需要两个？

**1. Webhook（后端到后端）**：

- 可靠通知，Stripe 直接和我们服务器通信
- 即使用户关闭浏览器，也会执行
- 用于更新数据库、发送邮件等后端逻辑

**2. successUrl（浏览器跳转）**：

- 告诉用户"支付成功了"
- 提供友好的 UI 展示

**如果只有 successUrl**：用户关闭了浏览器，支付成功了但你没收到通知，用户也没变成 VIP。

**如果只有 Webhook**：用户不知道支付结果，不知道自己有没有变成 VIP。

### ⚙️ 本地开发时的处理

开发环境没有公网域名，Stripe 无法回调本地 `localhost`。

解决方案：**Stripe CLI**（本地开发工具）

```bash
# 安装 Stripe CLI
# 转发 Webhook 事件到本地
stripe listen --forward-to localhost:8080/webhook/stripe

# 终端会输出 webhook secret
# ⚠️ You can get the webhook signing secret by running `stripe listen`
```

这样开发时，Stripe CLI 会把 Stripe 的回调转发到你的本地服务器。

## 二、为什么有两种支付成功方式

这是因为 **Stripe 支持两种不同的支付方式**，它们的到账时间不同：

### 💳 即时支付 vs 异步支付

| 事件                                       | 支付方式     | 到账时间     | 例子                                  |
| ------------------------------------------ | ------------ | ------------ | ------------------------------------- |
| `checkout.session.completed`               | **即时支付** | 几秒内       | 信用卡、借记卡、Apple Pay、Google Pay |
| `checkout.session.async_payment_succeeded` | **异步支付** | 几分钟到几天 | 银行转账、SEPA、ACH                   |

---

### 🏪 场景解释

#### 场景1：信用卡支付（即时）

```
用户选择 "Visa 卡" 
    │
    ▼
Stripe 直接授权 → 立即扣款成功
    │
    ▼
触发: checkout.session.completed ✅
    │
    ▼
立刻给用户开通 VIP
```

#### 场景2：银行转账支付（异步）

```
用户选择 "Bank Transfer (SEPA)"
    │
    ▼
Stripe 显示: "请在3天内转账 €199"
    │
    ▼
用户去自己银行转账...（可能2天后才到账）
    │
    ▼
Stripe 收到转账
    │
    ▼
触发: checkout.session.async_payment_succeeded ✅
    │
    ▼
给用户开通 VIP
```

### 🎯 代码处理

两种情况最终都调用同一个方法处理：

```java
case "checkout.session.completed":
    paymentService.handlePaymentSuccess(session);  // 相同逻辑
    break;
    
case "checkout.session.async_payment_succeeded":
    paymentService.handlePaymentSuccess(session);  // 相同逻辑
    break;
```

**为什么用同一个方法？** 因为最终结果一样 —— 支付成功了，用户都应该变成 VIP。

### 💡 形象比喻

```
即时支付 = 刷卡机直接显示 "支付成功"（当场知道结果）

异步支付 = 写支票，对方要等几天银行清算后才真正收到钱
           （当时不确定，要等通知）
```

**代码加两个分支是为了覆盖所有支付成功的情况**，确保不管用户用哪种方式支付，最终都能正确开通 VIP。
