### 2.一个用户有多个 SSE 吗？

错误！用户 A 生成文章，会建立一个 `taskId`  和 一个 `SSE` 链接，形如 `WebSocket` 建立一次链接，后续需要推送消息，就直接调用 SSE，避免频繁创建 SSE 对象加大延迟

只有在一方主动关闭或者出现超时、异常、错误的时候才会去关闭 SSE 链接

### 3.为什么用代理调用？

```java
getProxy().agent1GenerateTitleOptions(state);
```

同类内部的方法 A 调用被 AOP 拦截的方法 B 时：

1. 直接调用 B，AOP 不生效
2. 通过代理（Spring 提供的当前请求的 AOP 代理对象），AOP 生效

```java
private ArticleAgentService getProxy() {
    try {
        return (ArticleAgentService) AopContext.currentProxy();
        //                              ↑ 从 AopContext 获取当前 Bean 的代理对象
    } catch (IllegalStateException e) {
        return this;  // 降级：如果获取失败，用原始对象
    }
}
```

### 5.怎么推送给前端的呢？

```java
articleAgentService.executePhase1_GenerateTitles(state, message -> {
    handleAgentMessage(taskId, message, state);
});

public void executePhase1_GenerateTitles(ArticleState state, Consumer<String> streamHandler) {
    ...
	streamHandler.accept(SseMessageTypeEnum.AGENT1_COMPLETE.getValue());
    ...
}
```

利用**回调方法**完成对前端的推送

Consumer 是自带的注解，提供了一个 accept 方法，执行 accept 方法，就会回调执行传入的方法，所以会直接执行 Lambda 表达式

```java
// 形如JS
streamHandler = message -> { handleAgentMessage(taskId, message, state); }
```

### 6. 为什么又创建一个 ArticleSate 对象？

现在使用的方案是 MySQL，每个智能体使用的都是新创建的 articleState，但是因为是从数据库现查询的，间接就是共享了状态，共享了数据库的数据

如果将 articleState 存在内存中共享不合理，**服务器重启、连接断开、线程切换都会导致数据丢失**，然而每个智能体都是异步执行的，可能每个阶段都在不同的线程中，数据会丢失

使用 MySQL + Redis 方案，以实际业务看不太合理，因为加入 Redis 提高了复杂度，需要维护两个数据库的数据一致性。从业务看，如果不需要人工干预，想要智能体快速生成结果，使用 Redis 可以更快，但是现在的实际情况是需要用户手动确认，不需要秒级的完整生成
