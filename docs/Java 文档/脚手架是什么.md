## 一、脚手架的本质

**脚手架 = 标准化工程模板 + 自动化配置机制 + 最佳实践封装**

👉 本质目标：

> **把重复劳动工程化，把复杂配置自动化，把最佳实践组件化**

------

## 二、脚手架解决了什么问题

### 1. 工程痛点演进

#### 阶段一：手动开发（低效）

- 手动建目录
- 手动配依赖
- 手动写工具类
  👉 问题：重复 + 易错 + 不统一

------

#### 阶段二：复制项目（伪复用）

- Copy 老项目代码
  👉 问题：
- 冗余代码污染
- 版本不可控
- 技术债累积

------

#### 阶段三：脚手架（工程化）

- 一键生成项目
- 模块化能力复用

👉 本质升级：

> 从“代码复用” → “能力复用”

------

## 三、脚手架的两种形态（必须区分）

### 1️⃣ 项目级脚手架（Project Scaffold）

👉 作用：**生成项目骨架**

典型工具：

- Vue：`npm create vue`
- React：`create-react-app`
- Spring Boot：`Spring Initializr`

### 输出内容

- 目录结构
- 构建配置（pom.xml / package.json）
- 启动类
- 基础依赖

👉 本质：**初始化工程**

------

### 2️⃣ 模块级脚手架（Starter）

👉 作用：**提供某一类能力（缓存 / 日志 / 安全）**

典型命名：

```
xxx-framework-xxx-starter
```

### 特点

- 引入即生效
- 自动配置
- 面向接口使用

👉 本质：**能力插件化**

------

## 四、Spring Boot Starter 的底层原理

### 核心机制：自动装配（AutoConfiguration）

### 启动流程

```
引入 starter
    ↓
Spring Boot 启动
    ↓
读取：
META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
    ↓
加载配置类（@Configuration）
    ↓
注册 Bean
    ↓
注入容器
```

## 五、四大核心组成

### 1️⃣ 自动配置入口

```
META-INF/spring/...AutoConfiguration.imports
```

👉 作用：

- 告诉 Spring Boot 要加载哪些配置类

------

### 2️⃣ 配置层（Infrastructure）

例如：

- RedisConfig

👉 职责：

- 创建 RedisTemplate
- 配置序列化策略

------

### 3️⃣ 抽象接口层（API）

例如：

- `ICacheService`

👉 职责：

- 定义统一调用规范
- 隐藏底层实现

------

### 4️⃣ 实现层（Core Logic）

例如：

- `CacheService`

👉 职责：

- 封装缓存逻辑
- 处理高并发问题
