# 一、脚手架的本质（核心一句话）

**脚手架 = 标准化工程模板 + 自动化配置机制 + 最佳实践封装**

👉 本质目标：

> **把重复劳动工程化，把复杂配置自动化，把最佳实践组件化**

------

# 二、脚手架解决了什么问题

## 1. 工程痛点演进

### 阶段一：手动开发（低效）

- 手动建目录
- 手动配依赖
- 手动写工具类
  👉 问题：重复 + 易错 + 不统一

------

### 阶段二：复制项目（伪复用）

- Copy 老项目代码
  👉 问题：
- 冗余代码污染
- 版本不可控
- 技术债累积

------

### 阶段三：脚手架（工程化）

- 一键生成项目
- 模块化能力复用

👉 本质升级：

> 从“代码复用” → “能力复用”

------

# 三、脚手架的两种形态（必须区分）

## 1️⃣ 项目级脚手架（Project Scaffold）

👉 作用：**生成项目骨架**

典型工具：

- Vue：`npm create vue`
- React：`create-react-app`
- Spring Boot：Spring Initializr

### 输出内容

- 目录结构
- 构建配置（pom.xml / package.json）
- 启动类
- 基础依赖

👉 本质：**初始化工程**

------

## 2️⃣ 模块级脚手架（Starter）

👉 作用：**提供某一类能力（缓存 / 日志 / 安全）**

典型命名：

```
xxx-framework-xxx-starter
```

### 特点

- 引入即生效
- 自动配置
- 面向接口使用

👉 本质：**能力插件化**

------

# 四、Spring Boot Starter 的底层原理（重点）

## 核心机制：自动装配（AutoConfiguration）

### 启动流程

```
引入 starter
    ↓
Spring Boot 启动
    ↓
读取：
META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
    ↓
加载配置类（@Configuration）
    ↓
注册 Bean
    ↓
注入容器
```

------

## 四大核心组成（必须记住）

### 1️⃣ 自动配置入口

```
META-INF/spring/...AutoConfiguration.imports
```

👉 作用：

- 告诉 Spring Boot 要加载哪些配置类

------

### 2️⃣ 配置层（Infrastructure）

例如：

- RedisConfig

👉 职责：

- 创建 RedisTemplate
- 配置序列化策略

------

### 3️⃣ 抽象接口层（API）

例如：

- `ICacheService`

👉 职责：

- 定义统一调用规范
- 隐藏底层实现

------

### 4️⃣ 实现层（Core Logic）

例如：

- `CacheService`

👉 职责：

- 封装缓存逻辑
- 处理高并发问题

------

# 五、缓存脚手架的核心能力拆解（重点）

## 1. 缓存访问模型（模板模式）

```java
cacheService.getOne(
    key,
    expire,
    timeUnit,
    () -> dbQuery(),
    clazz
)
```

👉 本质：

> **缓存逻辑 + 业务查询解耦**

------

## 2. 核心机制解析

### ✅ 缓存命中

```
Redis → 返回 → 反序列化
```

------

### ✅ 缓存未命中

```
执行 supplier → 查询 DB → 写入缓存
```

------

### ✅ 双重检查锁（防击穿）

```java
if (cache miss) {
    synchronized {
        if (cache miss again) {
            query DB
        }
    }
}
```

👉 作用：

- 防止大量请求同时打 DB

------

## 3. 解决的典型问题

| 问题     | 解决方案                       |
| -------- | ------------------------------ |
| 缓存穿透 | null 控制 / 可扩展空值缓存     |
| 缓存击穿 | synchronized + 双检            |
| 缓存雪崩 | 过期时间控制（可扩展随机 TTL） |
| 序列化   | 统一 JSON                      |
| 批量操作 | pipeline + multiGet            |

------

# 六、设计思想（高级理解）

## 1️⃣ 模板方法模式（核心）

```java
getOne(key, supplier)
```

👉 固定流程：

- 查缓存
- 查 DB
- 写缓存

👉 可变部分：

- DB 查询逻辑（由 supplier 提供）

------

## 2️⃣ 依赖倒置（DIP）

```java
ICacheService ← CacheService
```

👉 好处：

- 可替换 Redis → Caffeine → 本地缓存

------

## 3️⃣ 开闭原则（OCP）

- 不修改 CacheService
- 通过扩展实现新功能

------

## 4️⃣ 约定优于配置

👉 用户只需：

- 引入依赖
- 使用接口

无需关心：

- Redis 配置
- 序列化细节

------

## 5️⃣ 条件装配（Spring Boot 精髓）

```java
@ConditionalOnClass(RedisTemplate.class)
```

👉 作用：

- 有 Redis 才加载缓存模块

------

# 七、使用者视角（实际开发流程）

## Step 1：引入依赖

```xml
<dependency>
    <groupId>xxx</groupId>
    <artifactId>cache-starter</artifactId>
</dependency>
```

------

## Step 2：自动完成

- RedisTemplate 自动创建
- CacheService 自动注入
- 配置自动生效

------

## Step 3：直接使用

```java
@Autowired
private ICacheService<User> cacheService;
```

👉 核心收益：

> **零配置使用缓存能力**

# 九、终极总结（建议背下来）

### 一句话版本

> 脚手架是通过自动配置和模块化设计，将通用能力标准化封装，使开发者只需关注业务逻辑的工程化解决方案。

------

### 架构层理解（加分点）

> 脚手架的本质是“基础设施下沉”，把缓存、日志、配置等横切关注点抽离为独立模块，实现业务系统的轻量化与可扩展性。
