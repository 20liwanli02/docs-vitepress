# 数据链路层设备 —— 集线器（Hub）

## 一、问题背景：多主机如何互联？

### 1️⃣ 两台主机

- 直接连接即可（点对点通信）
- 可以实现基本通信（如打游戏）

------

### 2️⃣ 多台主机问题

如果主机数量增加：

❌ 两两连接的问题：

- 连接复杂度高（O(n²)）
- 布线极其混乱
- 不可扩展

------

## 二、早期方案：总线型网络

### 结构特点

- 所有主机连接到一条“总线”上
- 数据在总线上广播

------

### ❗ 存在问题

1. **冲突（Collision）问题**
   - 多台主机同时发送 → 信号碰撞
2. **单点故障**
   - 总线断裂 → 整个网络瘫痪
3. **维护困难**
   - 更换总线成本高

------

### ✅ 解决冲突的方法

👉 使用 **CSMA/CD 协议**

（Carrier Sense Multiple Access with Collision Detection）

作用：

- 监听信道是否空闲
- 发送中检测冲突
- 冲突后重传

------

## 三、改进方案：集线器（Hub）

### 1️⃣ 基本概念

- 中文：集线器
- 英文：Hub
- 作用：连接多台主机

------

### 2️⃣ 网络结构

```text
        A
        |
        |
B —— Hub —— C
        |
        |
        D
```

👉 星型拓扑（逻辑上仍类似总线）

------

## 四、集线器的工作原理

### 1️⃣ 所属层次

👉 **物理层设备**

- 只处理“比特流”
- 不解析数据帧
- 不识别 MAC 地址

------

### 2️⃣ 核心特点

#### ✅ 无智能转发

- 不知道目标是谁
- 不学习地址
- 不做决策

------

#### ✅ 广播转发机制

👉 行为规则：

> 从一个端口收到数据 → 向其他所有端口转发

------

### 3️⃣ 转发流程示例

#### 场景：A → D

1. A 发送数据（包含目标 MAC：D）
2. 数据进入 Hub
3. Hub 执行：

```text
输入端口：1
输出端口：2、3、4（全部转发）
```

------

1. 各主机处理：

| 主机 | 行为              |
| ---- | ----------------- |
| B    | MAC 不匹配 → 丢弃 |
| C    | MAC 不匹配 → 丢弃 |
| D    | MAC 匹配 → 接收   |

------

### 4️⃣ 回包过程（D → A）

1. D 读取 A 的 MAC 地址（来自之前数据）
2. 构造数据帧：
   - 源 MAC：D
   - 目的 MAC：A
3. 发送到 Hub
4. Hub 再次广播
5. A 接收，其它主机丢弃

------

## 五、关键问题：冲突仍然存在

### ❗ 为什么？

因为 Hub 的广播机制：

- 所有设备共享同一信道
- 同时发送 → 冲突不可避免

------

### ✅ 结论

👉 使用 Hub 的网络：

- 仍然是 **共享介质网络**
- 必须使用 **CSMA/CD 协议**

------

## 六、集线器的优缺点

### ✅ 优点

- 结构简单
- 成本低
- 易于扩展（比总线好）

------

### ❌ 缺点（重点）

1. **无智能**
2. **广播风暴**
3. **冲突严重**
4. **带宽共享**
5. **安全性差（所有主机都能收到数据）**

------

## 七、核心总结（面试重点）

### ⭐ 一句话总结

> 集线器 = 物理层设备 + 广播转发 + 无智能

------

### ⭐ 对比关键点

| 特性         | 集线器 |
| ------------ | ------ |
| 工作层       | 物理层 |
| 是否识别 MAC | ❌      |
| 转发方式     | 广播   |
| 是否有冲突   | ✅      |
| 是否智能     | ❌      |

------
